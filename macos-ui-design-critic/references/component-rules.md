# Component Rules

## Buttons

Use a clear hierarchy:
- primary
- secondary
- tertiary / quiet
- destructive

Rules:
- one main visual primary per main area
- destructive actions must not compete with the main action
- weak actions should look weak
- avoid rows of equally loud buttons

## Sidebar

Sidebar is for:
- navigation
- scope switching
- categories
- collections

Do not overload it with dense settings forms.

## Toolbar

Toolbar should contain:
- frequent actions
- search
- filters
- sort
- view switching

Keep it light and scanable.

## Forms

Forms should be:
- vertically organized
- clearly labeled
- easy to scan
- easy to correct

Validation should appear close to the relevant field.

Avoid:
- too much helper text
- crowded side-by-side fields
- too many outlined boxes competing for attention

## Lists and tables

For information-dense views:
- keep row rhythm consistent
- make selection obvious
- reduce separator noise
- keep metadata distinct but not overpowering
- prefer contextual actions over always-visible action clutter

## Icons

Use one consistent system-style icon language.

Rules:
- icons should support labels, not replace them
- avoid mixing visual styles
- avoid excessive colorful icons
- use icons when they improve scanning speed
